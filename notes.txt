Причина ясна из логов: бот не может зайти в Postgres из-за неверного пароля для tg_user.

pq: password authentication failed for user "tg_user" = пароль в DATABASE_URL (или в env compose) не совпадает с паролем пользователя в самой БД.

Что проверить и исправить:

Посмотреть, что сейчас в env проекта на сервере
(без вывода токена в чат)
cd /opt/tg-expenses-bot
grep -E "POSTGRES_DB|POSTGRES_USER|POSTGRES_PASSWORD|BOT_TOKEN" .env
Проверить, какие env реально применились в compose
docker compose config | grep -E "POSTGRES_DB|POSTGRES_USER|POSTGRES_PASSWORD|DATABASE_URL"
Ключевой момент про Postgres в контейнере
Если том postgres_data уже был создан, смена POSTGRES_PASSWORD в .env не меняет пароль внутри уже инициализированной БД автоматически.
То есть обычно происходит так:

сначала БД поднялась с паролем A
потом в .env поставили B
бот пытается с B, а БД ждёт A → authentication failed.
Вариант A (сохранить данные, правильно для рабочего случая)
Выставить в .env тот пароль, с которым БД была создана изначально, затем:

docker compose up -d
docker compose logs -f bot
Вариант B (если данных не жалко, проще всего)
Сбросить БД-том и поднять заново с текущим паролем из .env:

docker compose down -v
docker compose up -d --build
Это удалит все расходы/категории в БД.

Если хочешь, можешь скинуть сюда вывод:

docker compose config | grep -E "POSTGRES_USER|POSTGRES_PASSWORD|DATABASE_URL"
docker compose ps и я точно скажу, какой из вариантов у тебя сейчас правильный.
